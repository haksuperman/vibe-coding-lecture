# Webtoon Orchestrator — 웹툰 제작 오케스트레이터

웹툰 제작 파이프라인을 **에이전트 팀**으로 조율한다. 실행 모드는 **에이전트 팀**(기본): 세계관·캐릭터 바이블을 전 단계가 끊임없이 참조하고 일관성 검수가 각 단계에 개입하므로, 팀 통신(SendMessage)·공유 작업목록(TaskCreate)·파일 공유(`_workspace/`)를 함께 쓴다.

## 팀 구성 (파이프라인 + 생성-검증)

| 에이전트 | subagent_type | 산출물 |
|---------|---------------|--------|
| story-architect | general-purpose | `00_world_bible.md`, `00_character_sheets.md` |
| scenario-writer | general-purpose | `01_scenario_ep{N}.md` |
| storyboard-director | general-purpose | `02_storyboard_ep{N}.md` |
| art-prompt-designer | general-purpose | `03_art_prompts_ep{N}.md` |
| webtoon-illustrator | general-purpose | `_workspace/images/cut{n}.png` (선택 단계) |
| consistency-reviewer | general-purpose | `qa_report_ep{N}.md` |

> **작화 실행(webtoon-illustrator)은 선택 단계다.** 기본 파이프라인은 작화 프롬프트까지 산출한다. 사용자가 "실제 이미지를 생성"하길 원하고 `OPENAI_API_KEY`가 준비됐을 때만 마지막에 추가한다. 유료 API이므로 컷 수·호출 횟수를 알리고 승인받은 뒤 실행한다.

> 모든 Agent 호출에 `model: "opus"`를 명시한다. 각 에이전트는 자신의 `.claude/agents/{name}.md` 정의를 따르고 동명의 스킬을 사용한다.

## Phase 0: 컨텍스트 확인 (먼저 실행)

`_workspace/` 상태로 실행 모드를 판별한다:
- **`_workspace/` 없음** → 초기 실행 (전체 파이프라인)
- **있음 + 사용자가 부분 수정 요청**("콘티만 다시", "3컷 외형 고쳐") → **부분 재실행** (해당 에이전트만 재호출)
- **있음 + 새 화/새 입력 요청** → 새 화 실행 (기획 산출물은 재사용, 화차 번호 증가)
- **있음 + 전면 재작업 요청** → 기존 `_workspace/`를 `_workspace_prev/`로 이동 후 초기 실행

부분 실행 범위(작화 프롬프트까지 갈지, 기획만 할지)는 사용자 선택("기획·설정 우선" 등)을 존중한다.

## Phase 1: 팀 구성 & 작업 할당

1. `TeamCreate`로 5인 팀 생성.
2. `TaskCreate`로 파이프라인 작업을 의존성과 함께 등록:
   `기획안승인 → 기획 → 시나리오 → 콘티 → 작화프롬프트` (순차 의존), `검수`는 각 단계에 후행.
3. 산출물은 `_workspace/`에 파일로 저장(파일 기반), 진행은 태스크로 추적(태스크 기반), 실시간 조율은 SendMessage(메시지 기반).

## Phase 1.5: 제작 기획안 승인 게이트 (본격 생성 전 필수)

상세 산출물을 만들기 전에 **방향을 사용자와 먼저 합의**한다. 잘못된 방향으로 전체 파이프라인을 돌리는 낭비를 막는 인간 승인 게이트다.

1. `story-architect`가 **간단한 제작 기획안**을 먼저 작성한다 → `_workspace/00_plan.md`
   - 로그라인(한 문장), 장르·톤·무드, 타깃 독자
   - 주요 캐릭터 3~5명 한 줄 소개(이름·역할·한 줄 개성)
   - 전체 연재 구조 개요 + 이번 작업 화의 화 구성(도입→절정→훅) 한 단락
2. 오케스트레이터가 기획안을 **사용자에게 제시하고 승인을 받는다**.
   - 승인 → Phase 2로 진행
   - 수정 요청 → `story-architect`가 기획안만 고쳐 재제시(상세 생성으로 넘어가지 않음)
3. 승인된 `00_plan.md`는 이후 모든 단계의 상위 기준이다. 세계관 바이블·캐릭터 시트는 이 기획안을 구체화한 것이어야 한다.

> 부분 재실행(이미 승인된 기획이 있고 "콘티만 다시" 등)에서는 이 게이트를 건너뛴다. 단, 사용자가 컨셉 자체를 바꾸면 게이트를 다시 통과한다.

## Phase 2: 파이프라인 실행 (점진적 QA 결합)

```
story-architect → [QA: 바이블 내부 정합]
      ↓
scenario-writer → [QA: 캐릭터 시트 ↔ 대사]
      ↓
storyboard-director → [QA: 시나리오 ↔ 콘티]
      ↓
art-prompt-designer → [QA: 콘티 ↔ 프롬프트, 외형 블록 ↔ 캐릭터 시트]
      ↓
(선택) webtoon-illustrator → 실제 PNG 생성  // 키 준비 + 사용자 승인 시에만
      ↓
[QA 최종 종합] → 사용자 보고
```

- 각 생성 에이전트는 완료 시 다음 에이전트와 `consistency-reviewer`에 SendMessage로 통지한다.
- 검수자는 **각 단계 직후** 해당 경계면을 즉시 검수한다(전체 끝나고 1회 아님).
- blocker 발견 시 해당 단계에서 멈추고 책임 에이전트가 수정 후 재검수로 닫은 다음 진행한다.

## 데이터 전달 프로토콜

- **파일 기반**: 모든 중간 산출물은 `_workspace/{phase}_{artifact}_ep{N}.md`. 최종 산출물도 `_workspace/`에 보존(감사 추적).
- **태스크 기반**: 작업 의존·진행상황은 TaskCreate/TaskUpdate.
- **메시지 기반**: 단계 완성 통지, QA 지적, 역질의는 SendMessage.
- 파일명 컨벤션: `00_world_bible.md`, `00_character_sheets.md`, `01_scenario_ep{N}.md`, `02_storyboard_ep{N}.md`, `03_art_prompts_ep{N}.md`, `qa_report_ep{N}.md`.

## 에러 핸들링

- 에이전트 1회 재시도 후 재실패 → 해당 산출물 없이 진행하고 최종 보고에 누락을 명시한다.
- 선행 산출물 부재 → 해당 에이전트는 멈추고 선행 에이전트 호출을 리더에 요청.
- 상충 데이터 → 삭제하지 않고 QA 리포트에 출처 병기.
- blocker QA → 진행 보류, 책임 에이전트 수정, 재검수 통과 후 재개.

## Phase 3: 종합 보고 & 피드백 수집

- 산출물 위치와 QA 요약을 사용자에게 보고한다.
- "결과에서 개선할 부분이 있나요? 팀 구성·워크플로우를 바꾸고 싶나요?"로 피드백 기회를 제공한다(강요 없음).
- 피드백은 유형별로 반영: 품질→스킬, 역할→에이전트 정의, 순서→이 오케스트레이터, 트리거 누락→description. 변경은 CLAUDE.md 변경 이력에 기록한다.

## 팀 크기
5인(중규모, 10~20 작업). 화당 컷 수가 많으면 작업을 씬 단위로 분할하되 팀원은 늘리지 않는다.

## 테스트 시나리오

**정상 흐름**: "마법 학원 로맨스 1화 만들어줘" → Phase 0(초기) → 팀 구성 → **Phase 1.5: 기획안(`00_plan.md`) 제시 → 사용자 승인** → 기획(바이블·캐릭터)→시나리오→콘티→작화프롬프트, 각 단계 QA 통과 → `_workspace/`에 산출물 + QA 리포트 → 보고.

**에러 흐름**: "이 화 콘티만 다시 해줘" (기획·시나리오 존재) → Phase 0(부분 재실행) → `storyboard-director`만 재호출(이전 콘티 읽고 수정) → QA가 시나리오↔콘티 재검수 → 작화 프롬프트는 콘티 변경 컷만 갱신 → 보고. 시나리오 파일이 없으면 콘티 단계가 멈추고 `scenario-writer` 선행을 요청.
